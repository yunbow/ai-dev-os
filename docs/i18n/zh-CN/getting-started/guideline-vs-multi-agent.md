# AI 代码审查：指南 vs 多智能体审查 — 使用指南

## 问题

> "我应该使用 AI Dev OS 指南，还是让多个 LLM 讨论我的代码？"

**答案：两者都用 — 但在不同层面。**

AI Dev OS 指南和多 LLM 审查不是替代关系。它们覆盖不同的质量维度，根据风险等级组合使用效果最佳。

## 核心洞察

```
质量
 ↑
 |          ┌─── 多 LLM 辩论（天花板效应）
 |         ╱     发现未知问题：边界情况、零日漏洞、设计权衡
 |        ╱
 |       ╱
 |  ════╪══════ 指南（地板效应）
 |      |       保证基线：命名、模式、安全检查清单、一致性
 |      |
 └──────┴──────→ 成本
```

- **指南提升下限** — 可复现、近零成本、知识随时间积累
- **多 LLM 辩论提升上限** — 探索未知问题，但成本高且不可复现

## 质量维度

| 质量维度 | 指南（AI Dev OS） | 多 LLM 辩论 | 主要负责方 |
|---|---|---|---|
| 合规性（编码规范） | ★★★★★ | ★★ | 指南 |
| 可维护性（命名、结构） | ★★★★★ | ★★ | 指南 |
| 正确性（逻辑缺陷） | ★★★ | ★★★★ | 多 LLM |
| 健壮性（边界情况） | ★★ | ★★★★ | 多 LLM |
| 安全性（漏洞） | ★★★ | ★★★★★ | 多 LLM（+ 指南） |
| 知识留存 | ★★★★★ | ★ | 指南 |
| 成本效率 | ★★★★★ | ★★ | 指南 |

## 基于任务的指南

| 任务 | 风险 | 推荐方法 | 估算成本 |
|---|---|---|---|
| CRUD 实现 | 低 | 仅指南（`/ai-dev-os-check`） | ~$0 |
| UI 组件 | 低 | 仅指南 | ~$0 |
| 业务逻辑 | 中 | 指南 + 单 LLM 审查（独立会话） | ~$0.04 |
| 重构 | 中 | 指南 + 测试套件 | ~$0 |
| API 设计 | 中高 | 指南 + 2 模型审查 | ~$0.40 |
| 安全相关代码 | 高 | 指南 + 3 模型辩论 | ~$0.80 |
| 支付/认证 | 极高 | 指南 + 3 模型辩论 + 人工审查 | ~$0.80+ |
| 架构决策 | 高 | 3 模型辩论（无唯一正确答案） | ~$0.80 |

## 推荐模型：指南优先，风险时辩论

```
阶段 1：代码生成
  AI 在上下文中加载 AI Dev OS 指南编写代码（CLAUDE.md）
  → 通过静态规则（10-15 条高影响力规则）确保基线质量
  → 成本：包含在正常 AI 编程中

阶段 2：提交前检查（/ai-dev-os-check）
  针对 30+ 条规则进行全面检查
  → 捕获合规性和已知模式违规
  → 成本：近零

阶段 3：PR 审查（基于风险）
  自动检测：安全/支付/认证文件的变更
  → 仅高风险 PR：触发跨模型审查
  → 成本：仅覆盖 10-20% 的 PR

阶段 4：知识反馈
  多 LLM 辩论发现的新模式 → 添加到 L3 指南
  → 下次同一问题以近零成本被捕获
  → 这就是Rule Harvesting的实践
```

## 为什么不对所有内容都使用多 LLM 辩论？

| 关注点 | 数据 |
|---|---|
| **成本** | 多智能体系统消耗的 token 是单智能体的 4-220 倍（ICLR 2025）。10 人团队对所有 PR 辩论 = 每月约 $16,000，而指南约为 $0 |
| **可复现性** | 相同代码 + 相同模型 = 每次不同结果。指南产生确定性检查 |
| **知识流失** | 辩论结果是易失的 — 上周的洞察不会延续到下次审查。指南会积累 |
| **采用门槛** | 不到 10% 的企业成功扩展多智能体系统。`npx ai-dev-os init` 第一天就能使用 |
| **误报** | LLM 系统性地过度纠正正确代码并错误归因缺陷原因（2026 年研究） |

## 为什么不能只用指南？

| 关注点 | 数据 |
|---|---|
| **迎合性** | LLM 有 56-62% 的概率附和用户期望（SycEval, AAAI 2025）。即使检查指南时，审查 LLM 也可能宽松地应用规则 — 约 40% 残余迎合风险 |
| **静态盲点** | 指南只能捕获已经写下的内容。新型漏洞、意外的边界情况和特定领域的问题需要探索 |
| **同一模型偏见** | 单一模型有一组偏见。它系统性地每次遗漏同类缺陷 |

## 反馈循环

最强大的洞察：**多 LLM 辩论是指南发现引擎。**

```
多 LLM 辩论发现新模式
  → 提取为 L3 指南（Rule Harvesting）
  → 添加到 ai-dev-os 规则
  → 下次：通过 /ai-dev-os-check 以零成本捕获
  → 辩论预算释放给下一个未知问题
```

随着时间推移，指南集扩展覆盖更多模式，辩论预算集中在真正新颖的问题上。这创造了良性循环。
