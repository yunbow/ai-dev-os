# 5 分钟了解 AI Dev OS — 框架比较

## 什么是 AI Dev OS？— AI 编程规则框架

一个将开发者隐性知识转化为显性的、可执行规则的框架，用于 AI 辅助编程。与工作流编排工具不同，AI Dev OS 将问题视为**知识管理**。

## 何时选择 AI Dev OS

**适合选择 AI Dev OS 的情况：**

- 希望规则能在工具更换时保留（Claude Code → Kiro → Cursor）
- 需要一个随时间改进的系统（Rule Harvesting + SECI 螺旋）
- 重视"为什么"而非"是什么" — 理解原则，而非仅仅遵循模板
- 跨多个项目工作，需要可复用的指南

**适合选择其他框架的情况：**

- 需要不带层级模型的轻量上下文文档（→ ContextKit）
- 需要单会话的工作流优化器（→ GSD）
- 需要面向大团队的多智能体编排（→ BMAD）
- 需要 GitHub 原生的规范管理（→ Spec Kit）

## 权衡

AI Dev OS 并非适用于所有情况：

- **前期投入**：前 30 分钟内看不到价值。规则需要逐步编写和打磨。
- **设置复杂度**：即使有 CLI，submodule 模型也需要 git 知识。不支持非 git 工作流。
- **规则维护**：规则需要定期审查。缺乏纪律性会导致规则过时。

## 关键差异化因素

### 1. 带有效期的规则

每条规则都有保质期。L1（哲学）持续 2-5 年。L4（工具配置）持续 2-4 个月。这防止了规则腐化。

### 2. 自下而上的规则发现

不要自上而下地编写规则。让 AI 生成代码，发现差距，从实际失败中提取规则。基于经验而非理论的规则。

### 3. 上下文"少即是多"

研究表明过多规则会降低 AI 输出质量。AI Dev OS 精选 3-5 条高影响力规则作为静态上下文，按需动态检查全部 30+ 条规则。

### 4. 哨兵规则

零违规的安全规则是*有效的* — 绝不能因"未使用"而删除。

## 设计哲学：Markdown 优先设计

AI Dev OS 有意完全由 Markdown 文件组成。这是有意为之的设计决策，而非限制：

- **透明性**：每条规则都是人类可读的。无需编译，无二进制格式，无不透明配置。
- **可 Fork 性**：任何人都可以 Fork、修改和重新分发规则，无需学习专有工具或 DSL。
- **工具无关性**：Markdown 可在任何编辑器、AI 工具、CI 系统中使用。无供应商锁定。

[CLI 工具](https://github.com/yunbow/ai-dev-os-cli)（`npx ai-dev-os init`）自动化设置，但底层模型仍是 git submodule — 规则以可审计、可比较差异的文本文件形式存在于您的仓库中。CLI 是可选的；始终支持手动设置。
