# Lifespan Layers — AI 코딩 규칙을 위한 4계층 모델

## 개요

AI Dev OS는 개발자 지식을 4개의 계층으로 구성하며, 각 계층은 고유한 수명과 추상 수준을 가집니다.

```text
┌──────────────────────────────────────────────────────┐
│ L1: 철학                          수명: 2-5년         │
│ "왜 이렇게 만드는가"                                   │
│ 도구와 언어를 초월하는 핵심 가치                         │
├──────────────────────────────────────────────────────┤
│ L2: 판단 기준                      수명: 1-3년         │
│ "어떻게 결정하는가"                                     │
│ 설계 및 아키텍처 원칙                                   │
├──────────────────────────────────────────────────────┤
│ L3: 가이드라인                     수명: 6-12개월       │
│ "무엇을 해야 하는가"                                    │
│ 구체적이고 검증 가능한 코딩 규칙                         │
├──────────────────────────────────────────────────────┤
│ L4: AI 프레임                      수명: 2-4개월        │
│ "어떻게 소통하는가"                                     │
│ 도구별 설정 및 워크플로                                  │
└──────────────────────────────────────────────────────┘
```

## 설계 원칙

### 의존성 규칙 (Clean Architecture에서 유래)

상위 계층은 하위 계층을 참조해서는 안 됩니다.

- L1은 도구 이름("Claude Code", "Cursor")이나 프레임워크 이름("Next.js", "FastAPI")을 포함해서는 안 됩니다
- L2는 프레임워크별 구현 세부사항을 포함해서는 안 됩니다
- L3은 프레임워크를 참조할 수 있지만 특정 도구 설정은 포함하면 안 됩니다
- L4는 정의상 도구별입니다

### 수명 = 추상 수준

계층이 추상적일수록 더 오래 살아남습니다. 이는 우연이 아닙니다 — 15개 고전 소프트웨어 공학 이론에서 관찰된 보편적 법칙입니다([classical-theories.md](../../theory/classical-theories.md) 참조).

### 각 계층은 질적으로 다릅니다

계층은 상세도의 수준이 아닙니다. 서로 다른 대상을 위해 서로 다른 목적을 가집니다 (Zachman Framework에 따름):

| 계층 | 목적 | 대상 |
|------|------|------|
| L1 | 팀 가치 정렬 | 모든 사람 |
| L2 | 아키텍처 결정 안내 | 아키텍트, 시니어 엔지니어 |
| L3 | 코딩 표준 강제 | 모든 개발자 |
| L4 | AI 도구 설정 | AI 도구 사용자 |

## Specificity Cascade

규칙이 충돌할 때, 특이성으로 해결합니다:

```text
1. [최고]   frameworks/[stack]/*   ← 가장 구체적
2. [높음]   common/*               ← 공통이지만 구체적
3. [중간]   project-specific/*     ← 프로젝트 컨텍스트
4. [낮음]   02_decision-criteria/* ← 추상적 기준
5. [최저]   01_philosophy/*        ← 가장 추상적
```

## Rule Harvesting

AI Dev OS는 하향식이 아닌 상향식으로 구축됩니다:

1. **코드 우선** — AI가 코드를 생성하게 합니다
2. **리뷰** — 출력과 기대 사이의 격차를 찾습니다
3. **규칙 수확** — 격차를 명시적 가이드라인으로 변환합니다 (L3)
4. **원칙 발견** — 규칙들에서 패턴을 찾습니다 (L2)
5. **철학 명문화** — 핵심 가치를 정의합니다 (L1)

이 접근법은 규칙이 가설적 이상이 아닌 실제 경험에 기반하도록 보장합니다.

## 도구 독립성

4계층 모델은 수명에 따라 관심사를 분리합니다:

- **L1-L3 (75%)** — 도구 독립적. Claude Code, Kiro, Cursor 등의 도구 간 마이그레이션에서도 유지됩니다.
- **L4 (25%)** — 도구별. 각 플러그인 리포지토리에서 구현됩니다.

이것이 AI Dev OS의 핵심 차별화 요소입니다: AI 도구를 전환해도 투자의 75%가 보존됩니다.

## Two-Tier Context Strategy（2단계 컨텍스트 전략）

모든 가이드라인을 항상 AI의 컨텍스트에 로드해야 하는 것은 아닙니다. 연구에 따르면 너무 많은 규칙은 AI 출력 품질을 저하시킬 수 있습니다.

**정적 컨텍스트** (CLAUDE.md / .cursorrules / AGENTS.md):

- 모든 대화에서 AI가 로드
- **10-15개의 고영향 가이드라인만** 포함해야 합니다 (보안, 에러 처리, 네이밍, 프로젝트 구조, 프레임워크 개요)
- 수동으로 선별 — 위반 빈도를 기준으로 자동 제거하지 마세요 (보안과 같은 저빈도 규칙은 존재하기 때문에 정확히 작동하는 것일 수 있습니다)

**동적 검사** (플러그인 스킬/규칙):

- 요청 시 호출 (`/ai-dev-os-check`, `/ai-dev-os-scan`, `/ai-dev-os-review`)
- **모든 가이드라인** (30개 이상)에 대해 포괄적으로 검증
- 코드 생성 중이 아닌 집중된 리뷰 작업에서 AI가 처리하므로 주의력 분산 없음

이 2단계 접근법은 두 가지 문제를 모두 해결합니다:

- "너무 많은 규칙이 품질을 저하시킨다" → 정적 컨텍스트는 선별되고 집중됨
- "중요한 규칙을 놓쳐서는 안 된다" → 동적 검사가 모든 것을 커버

## 감시병 규칙

> **위반의 부재는 보호의 존재를 증명하는 것이지, 필요의 부재를 증명하는 것이 아닙니다.**

위반이 없는 보안 규칙은 *작동하고 있는 것*입니다. "한 번도 발동되지 않았다"고 제거하면 즉시 회귀를 일으킵니다. 이 원칙은 모든 고위험 규칙(보안, 인증, 데이터 무결성)에 적용됩니다:

- 낮은 위반 빈도를 기준으로 규칙을 자동 제거하지 마세요
- 문제를 방지해 온 규칙의 우선순위를 낮추지 마세요
- 정적 컨텍스트를 선별할 때, 위반 횟수와 관계없이 보안 및 인증 규칙은 항상 유지하세요
