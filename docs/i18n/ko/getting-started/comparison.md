# 5분 만에 보는 AI Dev OS — 프레임워크 비교

## AI Dev OS란? — AI 코딩 규칙 프레임워크

개발자의 암묵적 지식을 AI 지원 코딩을 위한 명시적이고 강제 가능한 규칙으로 변환하는 프레임워크입니다. 워크플로 오케스트레이션 도구와 달리, AI Dev OS는 이 문제를 **지식 관리**로 접근합니다.

## AI Dev OS를 선택해야 할 때

**다음에 해당하면 AI Dev OS를 선택하세요:**
- 도구 변경(Claude Code → Kiro → Cursor)에서도 살아남는 규칙을 원할 때
- 시간이 지남에 따라 개선되는 시스템이 필요할 때 (Rule Harvesting + SECI 나선)
- "무엇을" 보다 "왜"를 중시할 때 — 템플릿을 따르는 것이 아닌 원칙을 이해
- 여러 프로젝트에서 재사용 가능한 가이드라인을 원할 때

**다른 것을 선택해야 할 때:**
- 계층 모델 없이 가벼운 컨텍스트 문서를 원할 때 (→ ContextKit)
- 단일 세션 워크플로 최적화를 원할 때 (→ GSD)
- 대규모 팀을 위한 멀티 에이전트 오케스트레이션이 필요할 때 (→ BMAD)
- GitHub 네이티브 명세 관리를 원할 때 (→ Spec Kit)

## 트레이드오프

AI Dev OS가 모든 상황에 적합한 것은 아닙니다:

- **초기 투자**: 처음 30분 안에 가치를 느끼기 어렵습니다. 규칙은 시간을 들여 작성하고 개선해야 합니다.
- **설정 복잡도**: CLI가 있더라도 서브모듈 모델은 git 지식을 필요로 합니다. git을 사용하지 않는 워크플로는 지원되지 않습니다.
- **규칙 유지보수**: 규칙은 주기적인 검토가 필요합니다. 규율이 없으면 낡아집니다.

## 핵심 차별화 요소

### 1. 유효기한이 있는 규칙
모든 규칙에는 수명이 있습니다. L1(철학)은 2-5년 지속됩니다. L4(도구 설정)는 2-4개월 지속됩니다. 이는 규칙의 노후화를 방지합니다.

### 2. 상향식 규칙 발견
규칙을 하향식으로 작성하지 마세요. AI가 코드를 생성하게 하고, 격차를 찾고, 실제 실패에서 규칙을 추출하세요. 이론이 아닌 경험에 기반한 규칙입니다.

### 3. 컨텍스트의 "적을수록 좋다"
연구에 따르면 규칙이 너무 많으면 AI 출력 품질이 저하됩니다. AI Dev OS는 정적 컨텍스트에 10-15개의 고영향 규칙을 선별하고, 요청 시 30개 이상을 동적으로 검사합니다.

### 4. 감시병 규칙
위반이 없는 보안 규칙은 *작동하고 있는 것*입니다 — "사용되지 않는다"고 절대 제거하지 마세요.

## 설계 철학: Markdown 기반 설계

AI Dev OS는 의도적으로 전체가 Markdown 파일로 구성되어 있습니다. 이것은 제한이 아니라 의도적인 설계 결정입니다:

- **투명성**: 모든 규칙이 사람이 읽을 수 있습니다. 컴파일 불필요, 바이너리 형식 없음, 불투명한 설정 없음.
- **포크 가능성**: 누구나 규칙을 포크하고 수정하고 재배포할 수 있습니다. 독점 도구나 DSL 학습이 필요 없습니다.
- **도구 독립성**: Markdown은 모든 편집기, AI 도구, CI 시스템에서 작동합니다. 벤더 종속 없음.

[CLI 도구](https://github.com/yunbow/ai-dev-os-cli)(`npx ai-dev-os init`)는 설정을 자동화하지만, 기반 모델은 여전히 git submodule입니다 — 규칙은 감사 가능하고 diff 확인 가능한 텍스트 파일로 리포지토리에 존재합니다. CLI는 선택 사항이며, 수동 설정은 항상 지원됩니다.
