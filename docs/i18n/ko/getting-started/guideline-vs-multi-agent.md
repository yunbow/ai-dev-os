# AI 코드 리뷰: 가이드라인 vs 멀티 에이전트 리뷰 — 언제 무엇을 사용할 것인가

## 질문

> "AI Dev OS 가이드라인을 사용해야 할까, 아니면 여러 LLM이 내 코드를 토론하게 해야 할까?"

**답: 둘 다 — 하지만 다른 계층에서.**

AI Dev OS 가이드라인과 멀티 LLM 리뷰는 대안이 아닙니다. 서로 다른 품질 차원을 다루며, 위험 수준에 따라 결합했을 때 가장 효과적입니다.

## 핵심 통찰

```
품질
 ↑
 |          ┌─── 멀티 LLM 토론 (천장 효과)
 |         ╱     미지의 이슈 발견: 엣지 케이스, 제로데이, 설계 트레이드오프
 |        ╱
 |       ╱
 |  ════╪══════ 가이드라인 (바닥 효과)
 |      |       기본선 보장: 네이밍, 패턴, 보안 체크리스트, 일관성
 |      |
 └──────┴──────→ 비용
```

- **가이드라인은 바닥을 올립니다** — 재현 가능하고, 비용이 거의 없으며, 지식이 시간이 지남에 따라 축적됩니다
- **멀티 LLM 토론은 천장을 올립니다** — 미지의 문제를 탐색하지만, 비용이 높고 재현이 불가능합니다

## 품질 차원

| 품질 차원 | 가이드라인 (AI Dev OS) | 멀티 LLM 토론 | 주 담당 |
|-----------|----------------------|---------------|---------|
| 적합성 (코딩 표준) | ★★★★★ | ★★ | 가이드라인 |
| 유지보수성 (네이밍, 구조) | ★★★★★ | ★★ | 가이드라인 |
| 정확성 (로직 버그) | ★★★ | ★★★★ | 멀티 LLM |
| 견고성 (엣지 케이스) | ★★ | ★★★★ | 멀티 LLM |
| 보안 (취약점) | ★★★ | ★★★★★ | 멀티 LLM (+ 가이드라인) |
| 지식 보존 | ★★★★★ | ★ | 가이드라인 |
| 비용 효율 | ★★★★★ | ★★ | 가이드라인 |

## 작업별 가이드

| 작업 | 위험도 | 권장 접근법 | 예상 비용 |
|------|--------|------------|-----------|
| CRUD 구현 | 낮음 | 가이드라인만 (`/ai-dev-os-check`) | ~$0 |
| UI 컴포넌트 | 낮음 | 가이드라인만 | ~$0 |
| 비즈니스 로직 | 중간 | 가이드라인 + 단일 LLM 리뷰 (별도 세션) | ~$0.04 |
| 리팩토링 | 중간 | 가이드라인 + 테스트 스위트 | ~$0 |
| API 설계 | 중-고 | 가이드라인 + 2모델 리뷰 | ~$0.40 |
| 보안 관련 코드 | 높음 | 가이드라인 + 3모델 토론 | ~$0.80 |
| 결제 / 인증 | 치명적 | 가이드라인 + 3모델 토론 + 사람 리뷰 | ~$0.80+ |
| 아키텍처 결정 | 높음 | 3모델 토론 (정답이 하나가 아님) | ~$0.80 |

## 권장 모델: 가이드라인 우선, 위험 기반 토론

```
1단계: 코드 생성
  AI가 AI Dev OS 가이드라인이 컨텍스트에 로드된 상태로 코드 작성 (CLAUDE.md)
  → 정적 규칙을 통한 기본 품질 (10-15개 고영향 규칙)
  → 비용: 일반 AI 코딩에 포함

2단계: Pre-Commit 검사 (/ai-dev-os-check)
  30개 이상 규칙에 대한 포괄적 검사
  → 적합성 및 알려진 패턴 위반 포착
  → 비용: 거의 무료

3단계: PR 리뷰 (위험 기반)
  자동 감지: 보안/결제/인증 파일 변경
  → 고위험 PR만: 크로스 모델 리뷰 트리거
  → 비용: PR의 10-20%에만 해당

4단계: 지식 피드백
  멀티 LLM 토론에서 발견된 새로운 패턴 → L3 가이드라인에 추가
  → 다음번에는 동일 이슈가 거의 무료로 포착됨
  → Rule Harvesting의 실천
```

## 왜 모든 것에 멀티 LLM 토론을 사용하지 않는가?

| 우려 사항 | 데이터 |
|-----------|--------|
| **비용** | 멀티 에이전트 시스템은 싱글 에이전트보다 4-220배 더 많은 토큰을 소비합니다 (ICLR 2025). 10명 개발자 팀의 경우, 전체 PR 토론 = 월 ~$16,000 vs 가이드라인 ~$0 |
| **재현성** | 동일 코드 + 동일 모델 = 매번 다른 결과. 가이드라인은 결정론적 검사를 생산합니다 |
| **지식 손실** | 토론 결과는 휘발성입니다 — 지난주의 인사이트가 다음 리뷰에 전달되지 않습니다. 가이드라인은 축적됩니다 |
| **도입 장벽** | 기업의 10% 미만이 멀티 에이전트 시스템을 성공적으로 확장합니다. `npx ai-dev-os init`은 첫날부터 작동합니다 |
| **오탐** | LLM은 올바른 코드를 체계적으로 과잉 수정하고 결함 원인을 잘못 귀속합니다 (2026 연구) |

## 왜 가이드라인만으로는 안 되는가?

| 우려 사항 | 데이터 |
|-----------|--------|
| **아첨 편향** | LLM은 56-62%의 경우 사용자 기대에 동의합니다 (SycEval, AAAI 2025). 가이드라인을 검사할 때도 검토 LLM이 규칙을 관대하게 적용할 수 있습니다 — ~40% 잔여 아첨 위험 |
| **정적 사각지대** | 가이드라인은 문서화된 것만 포착합니다. 새로운 취약점, 예상치 못한 엣지 케이스, 도메인별 이슈는 탐색이 필요합니다 |
| **동일 모델 편향** | 단일 모델은 하나의 편향 세트를 가집니다. 매번 동일한 유형의 버그를 체계적으로 놓칩니다 |

## 피드백 루프

가장 강력한 통찰: **멀티 LLM 토론은 가이드라인 발견 엔진입니다**.

```
멀티 LLM 토론이 새로운 패턴 발견
  → L3 가이드라인으로 추출 (Rule Harvesting)
  → ai-dev-os 규칙에 추가
  → 다음번에: /ai-dev-os-check로 무료 포착
  → 토론 예산이 다음 미지의 이슈에 집중
```

시간이 지남에 따라 가이드라인 세트가 더 많은 패턴을 커버하도록 성장하고, 토론 예산은 진정으로 새로운 이슈에 집중됩니다. 이는 선순환을 만듭니다.
