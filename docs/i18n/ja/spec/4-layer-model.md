# Lifespan Layers — AI コーディングルールのための4層モデル

## 概要

AI Dev OS は開発者の知識を4つの層に整理し、それぞれ異なるライフスパンと抽象度を持たせています。

```
┌──────────────────────────────────────────────────────┐
│ L1: Philosophy                    Lifespan: 2-5 years │
│ "Why we build this way"                               │
│ Core values that transcend tools and languages         │
├──────────────────────────────────────────────────────┤
│ L2: Decision Criteria             Lifespan: 1-3 years │
│ "How to decide"                                       │
│ Design and architecture principles                     │
├──────────────────────────────────────────────────────┤
│ L3: Guidelines                    Lifespan: 6-12 months│
│ "What to do"                                          │
│ Concrete, verifiable coding rules                      │
├──────────────────────────────────────────────────────┤
│ L4: AI Frames                     Lifespan: 2-4 months │
│ "How to communicate"                                  │
│ Tool-specific configurations and workflows             │
└──────────────────────────────────────────────────────┘
```

## 設計原則

### 依存性ルール（Clean Architecture より）

上位層は下位層を参照してはなりません。

- L1 にはツール名（"Claude Code"、"Cursor"）やフレームワーク名（"Next.js"、"FastAPI"）を含めてはならない
- L2 にはフレームワーク固有の実装詳細を含めてはならない
- L3 はフレームワークを参照してもよいが、特定のツール設定は含めてはならない
- L4 は定義上、ツール固有である

### ライフスパン = 抽象度

抽象度が高い層ほど、長く存続します。これは偶然ではなく、15の古典的ソフトウェア工学理論にわたって観察される普遍的な法則です（[classical-theories.md](../theory/classical-theories.md) を参照）。

### 各層は質的に異なる

層は詳細度のレベルではありません。Zachman Framework に基づき、異なる目的を異なる対象者に対して果たします。

| 層 | 目的 | 対象者 |
|-------|---------|----------|
| L1 | チームの価値観を揃える | 全員 |
| L2 | アーキテクチャ上の判断を導く | アーキテクト、シニアエンジニア |
| L3 | コーディング標準を徹底する | 全開発者 |
| L4 | AIツールを設定する | AIツールユーザー |

## Specificity Cascade

ルールが競合した場合、詳細度で解決します。

```
1. [Highest] frameworks/[stack]/*   ← Most specific
2. [High]    common/*               ← Common but concrete
3. [Medium]  project-specific/*     ← Project context
4. [Low]     02_decision-criteria/* ← Abstract criteria
5. [Lowest]  01_philosophy/*        ← Most abstract
```

## Rule Harvesting

AI Dev OS はトップダウンではなく、ボトムアップで構築されます。

1. **まずコードを書く** — AIにコードを生成させる
2. **レビューする** — 出力と期待値のギャップを見つける
3. **ルールを収穫する** — ギャップを明示的なガイドライン（L3）に変換する
4. **原則を発見する** — ルール間のパターンを見出す（L2）
5. **哲学を言語化する** — コアバリューを定義する（L1）

このアプローチにより、ルールは仮説的な理想ではなく、実体験に基づいたものになります。

## ツール非依存性

Lifespan Layers モデルはライフスパンごとに関心を分離します。

- **L1〜L3（75%）** — ツール非依存。Claude Code、Kiro、Cursor などの間の移行にも耐えます。
- **L4（25%）** — ツール固有。各プラグインリポジトリで実装されます。

これが AI Dev OS の最大の差別化要因です。AIツールを切り替えても、投資の75%が保持されます。

## Two-Tier Context Strategy（2層コンテキスト戦略）

すべてのガイドラインを常にAIのコンテキストに読み込むべきではありません。ルールが多すぎるとAIの出力品質が低下することが研究で示されています。

**静的コンテキスト**（CLAUDE.md / .cursorrules / AGENTS.md）:
- AIが全会話で読み込む
- **影響の大きい10〜15のガイドライン**のみ含めるべき（セキュリティ、エラーハンドリング、命名規則、プロジェクト構成、フレームワーク概要）
- 手動でキュレーションする — 違反頻度に基づいて自動削除してはならない（セキュリティのような低頻度ルールは、まさにそこに存在しているから機能している可能性がある）

**動的チェック**（プラグインのskills/rules）:
- オンデマンドで呼び出す（`/ai-dev-os-check`、`/ai-dev-os-scan`、`/ai-dev-os-review`）
- **すべてのガイドライン**（30以上）を包括的に検証する
- コード生成中ではなく、集中的なレビュータスクとしてAIが処理するため、注意力の分散が起きない

この2層アプローチは両方の問題を解決します。
- 「ルールが多すぎると品質が低下する」→ 静的コンテキストはキュレーションされ焦点が絞られている
- 「重要なルールを見落としてはならない」→ 動的チェックがすべてをカバーする

## Sentinel ルール

> **違反がないことは、保護が機能していることの証明であり、必要性がないことの証明ではない。**

違反ゼロのセキュリティルールは*機能している*のです。「一度も発動しない」からといって削除すれば、すぐにリグレッションが発生します。この原則はすべての高リスクルール（セキュリティ、認証、データ整合性）に適用されます。

- 低い違反頻度に基づいてルールを自動削除してはならない
- 問題を防止してきたルールの優先度を下げてはならない
- 静的コンテキストをキュレーションする際は、違反回数に関係なくセキュリティと認証のルールを常に維持すること
