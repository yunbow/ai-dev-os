# AI コードレビュー: ガイドライン vs マルチエージェントレビュー — 使い分けガイド

## 問い

> 「AI Dev OS のガイドラインを使うべきか、それとも複数のLLMにコードを議論させるべきか？」

**回答: 両方。ただしレイヤーが違う。**

AI Dev OS のガイドラインとマルチLLMレビューは代替関係ではない。異なる品質次元に作用するため、リスクレベルに応じて組み合わせるのが最善。

## 核心的な洞察

```text
品質
 ↑
 |          ┌─── マルチLLM議論（天井効果）
 |         ╱     未知の問題を発見: エッジケース、ゼロデイ、設計トレードオフ
 |        ╱
 |       ╱
 |  ════╪══════ ガイドライン（床効果）
 |      |       ベースラインを保証: 命名、パターン、セキュリティチェック、一貫性
 |      |
 └──────┴──────→ コスト
```

- **ガイドラインは品質の床を引き上げる** — 再現可能、コストほぼゼロ、知識が蓄積される
- **マルチLLM議論は品質の天井を押し上げる** — 未知の問題を探索できるが、高コストで再現性がない

## 品質次元別の有効性

| 品質次元 | ガイドライン (AI Dev OS) | マルチLLM議論 | 主担当 |
|---|---|---|---|
| 準拠性（コーディング規約） | ★★★★★ | ★★ | ガイドライン |
| 保守性（命名・構造） | ★★★★★ | ★★ | ガイドライン |
| 正確性（ロジックバグ） | ★★★ | ★★★★ | マルチLLM |
| 堅牢性（エッジケース） | ★★ | ★★★★ | マルチLLM |
| 安全性（脆弱性） | ★★★ | ★★★★★ | マルチLLM（+ ガイドライン） |
| 知識蓄積性 | ★★★★★ | ★ | ガイドライン |
| 費用対効果 | ★★★★★ | ★★ | ガイドライン |

## タスク別使い分けガイド

| タスク | リスク | 推奨アプローチ | 概算コスト/回 |
|---|---|---|---|
| CRUD実装 | 低 | ガイドラインのみ（`/ai-dev-os-check`） | ~$0 |
| UIコンポーネント | 低 | ガイドラインのみ | ~$0 |
| ビジネスロジック | 中 | ガイドライン + 単一LLM別セッションレビュー | ~$0.04 |
| リファクタリング | 中 | ガイドライン + テストスイート | ~$0 |
| API設計 | 中〜高 | ガイドライン + 異種2モデルレビュー | ~$0.40 |
| セキュリティ関連 | 高 | ガイドライン + 異種3モデル議論 | ~$0.80 |
| 決済・認証 | 極高 | ガイドライン + 異種3モデル議論 + 人間レビュー | ~$0.80+ |
| アーキテクチャ設計 | 高 | 異種3モデル議論（正解のないトレードオフ判断） | ~$0.80 |

## 推奨モデル: 「ガイドライン・ファースト、ディベート・オン・リスク」

```text
Phase 1: コード生成
  AI Dev OS ガイドラインをコンテキストに含めてコード生成（CLAUDE.md）
  → 静的ルール（3-5の高インパクトルール）でベースライン品質確保

Phase 2: コミット前チェック（/ai-dev-os-check）
  30以上のルールで網羅的チェック
  → 準拠性・既知パターン違反を検出

Phase 3: PRレビュー（リスクベース）
  セキュリティ/決済/認証ファイルの変更を自動検出
  → 高リスクPR（10-20%）のみクロスモデルレビューを発動

Phase 4: 知識のフィードバック
  マルチLLM議論で発見された新パターン → L3ガイドラインに追加
  → 次回からは同じ問題を /ai-dev-os-check でゼロコスト検出
  → これが Rule Harvesting の実践
```

## なぜ全てにマルチLLM議論を使わないのか？

| 懸念 | データ |
|---|---|
| **コスト** | マルチエージェントは単一比で4-220倍のトークン消費（ICLR 2025）。10人チームで全PRに適用すると月額約$16,000 |
| **再現性** | 同じコード + 同じモデル = 毎回異なる結果。ガイドラインは決定論的 |
| **知識の喪失** | 議論結果は揮発的。先週の知見は次のレビューに引き継がれない |
| **導入障壁** | マルチエージェントのスケール成功率は10%未満。`npx ai-dev-os init` は即日稼働 |

## なぜガイドラインだけでは不十分なのか？

| 懸念 | データ |
|---|---|
| **迎合性** | LLMはユーザーの期待に56-62%同調する（SycEval, AAAI 2025）。ガイドラインチェック時にも約40%の残存迎合リスク |
| **静的な盲点** | ガイドラインは書かれたものしか検出できない。未知の脆弱性や想定外のエッジケースには探索が必要 |
| **同一モデルバイアス** | 単一モデルは同じ種類のバグを体系的に見逃す |

## フィードバックループ

最も重要な洞察: **マルチLLM議論はガイドライン発見エンジンである。**

```text
マルチLLM議論で新パターン発見
  → L3ガイドラインとして抽出（Rule Harvesting）
  → ai-dev-os ルールに追加
  → 次回: /ai-dev-os-check でゼロコスト検出
  → 議論の予算は次の未知の問題に集中
```

時間の経過とともにガイドラインのカバレッジが広がり、議論の予算は真に未知の問題に集中できる。これが好循環を生む。
