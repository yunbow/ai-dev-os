# AI Dev OS を5分で理解する -- フレームワーク比較

## AI Dev OS とは? — AI コーディングルールフレームワーク

開発者の暗黙知を、AIによるコーディング支援で使える明示的かつ強制力のあるルールに変換するためのフレームワークです。ワークフローオーケストレーションツールとは異なり、AI Dev OS は問題を**ナレッジマネジメント**として捉えます。

## AI Dev OS を選ぶべきとき

**AI Dev OS が適しているケース:**

- ツールの変更に耐えるルールが欲しい（Claude Code → Kiro → Cursor）
- 時間とともに改善されるシステムが必要（Rule Harvesting + SECIスパイラル）
- 「何を」より「なぜ」を重視する -- テンプレートに従うだけでなく原則を理解したい
- 複数のプロジェクトをまたいで再利用可能なガイドラインが欲しい

**他のフレームワークが適しているケース:**

- 単一セッションのワークフロー最適化が目的（→ GSD）
- 大規模チーム向けのマルチエージェントオーケストレーションが必要（→ BMAD）
- GitHub ネイティブな仕様管理が欲しい（→ Spec Kit）

## 主な差別化要素

### 1. 有効期限付きルール

すべてのルールに寿命があります。L1（設計思想）は2-5年、L4（ツール設定）は2-4ヶ月です。これによりルールの陳腐化を防ぎます。

### 2. ボトムアップのルール発見

ルールをトップダウンで書くのではなく、AIにコードを生成させ、ギャップを見つけ、実際の失敗からルールを抽出します。理論ではなく経験に基づいたルールです。

### 3. コンテキストは「少ないほど良い」

研究によると、ルールが多すぎるとAIの出力品質が低下します。AI Dev OS は静的コンテキストとして影響の大きい3-5のルールを厳選し、30以上の全ルールはオンデマンドで動的にチェックします。

### 4. センチネルルール

違反件数がゼロのセキュリティルールは*機能している*証拠です。「使われていない」という理由で削除してはいけません。

## 設計思想: Markdown による設計

AI Dev OS は意図的にすべて Markdown ファイルで構成されています。これは制限ではなく、意図的な設計判断です:

- **透明性**: すべてのルールが人間に読める。コンパイル不要、バイナリフォーマット不要、不透明な設定不要。
- **フォーク可能性**: 誰でもルールをフォーク、修正、再配布可能。独自ツールやDSLの学習は不要。
- **ツール非依存性**: Markdown はどのエディタ、AIツール、CIシステムでも動作する。ベンダーロックインなし。

[CLI ツール](https://github.com/yunbow/ai-dev-os-cli)（`npx ai-dev-os init`）はセットアップを自動化しますが、基盤となるモデルは git submodule です — ルールはリポジトリ内の監査可能で差分確認可能なテキストファイルとして存在します。CLI はオプションであり、手動セットアップも常にサポートされます。
